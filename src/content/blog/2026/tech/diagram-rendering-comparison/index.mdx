给博客加流程图，看起来是个小需求——Markdown 里画几笔、构建时变成图片、页面打开就能看。但实际做下来，踩了四个方案的坑，最后发现真正的问题不在"能不能渲染"，而在渲染结果能不能**稳定、一致、不让人分心**。

这篇文章记录一下整个过程：试了哪四种方案、每种方案的取舍是什么、为什么最后觉得 D2 是最先进的选择。

## 问题的起点：` ```mermaid ` 代码块怎么变成图

需求本身很清楚：在 MDX 文章里写 mermaid 代码块，构建时自动渲染成图表，亮色和暗色模式下都能正常显示。

如果只是"能渲染"，选一个 mermaid 的 npm 包、配进 Astro 的 markdown 插件链就完事了。但很快就发现，mermaid 在这个场景下有一系列问题——不是功能缺失，而是渲染质量、样式控制和工程复杂度之间的权衡。

下面按尝试的顺序，逐个拆开四种方案。

## 方案一：客户端 mermaid 渲染

第一个想法是最直觉的：页面加载后，用 JS 找到所有 mermaid 代码块，调 `mermaid.render()` 替换成 SVG。

这个方案的实现很简单。一个 Astro 组件，从 CDN 加载 mermaid.js，`querySelectorAll` 找到所有代码块，逐个渲染。

但很快碰到三个问题。

第一个是选择器不匹配。Astro 用 Shiki 做语法高亮，mermaid 代码块被渲染成 `<div class="astro-code" data-language="mermaid">`，不是 `<code class="language-mermaid">`。修复选择器后能跑了，但这暴露了一个根本问题：客户端渲染对构建工具的 HTML 结构有隐式依赖，换个高亮器就可能失效。

第二个是文字裁切。多行文字的节点——比如「Level 3: Microcompact / 缓存热: cache_edits / 缓存冷: 直接替换」——在 mermaid 的 `base` 主题下，节点高度算不准，底部文字直接被切掉。试了 expand viewBox、调 padding、改 CSS overflow、渲染后 patch SVG——都没根治。因为 `base` 主题用 `foreignObject` 渲染文字，浏览器 HTML 布局和 SVG 布局之间的时序问题导致节点尺寸被低估。

第三个是亮暗模式切换。客户端渲染时拿到的是当时主题的颜色，切换主题后要么用 CSS filter 反转，要么监听 DOM 变化后全部重绘。前者简单但有色彩偏差，后者有闪烁。

**为什么没选**：文字裁切是硬伤，而且客户端渲染引入了不必要的运行时依赖和首屏闪烁。

## 方案二：服务端 Playwright 渲染

第二个想法是把渲染提前到构建时。写了一个 remark 插件，在 AST 里找到 mermaid 代码块，用 Playwright 启动一个 headless Chromium，打开一个页面跑 mermaid，等 SVG 生成后取回、嵌入 HTML。

这个方案解决了文字裁切问题——真实浏览器渲染，节点尺寸完全正确。

但引入了新的问题。构建时间从约 12 秒涨到约 55 秒，每个 mermaid 图都要启动一个独立的 Playwright page。本地需要安装 Playwright + Chromium（约 500MB）。Vercel 部署也需要在构建环境中预装 Playwright 浏览器。

样式控制也是一个问题。`themeCSS` 能覆盖大部分节点样式，但 edge label 用 `foreignObject` 渲染，SVG CSS 对内部 HTML 元素不生效。试了 `htmlLabels: false` ——edge label 样式统一了，但 `themeCSS` 里有些规则仍然被 mermaid 默认样式覆盖，某些节点类型会出现不一致的颜色。

**为什么没选**：构建时间翻四倍，依赖太重，样式仍然不够可控。

## 方案三：Kroki API

第三个想法是绕过本地渲染，交给外部服务。Kroki 是一个开源的图表渲染 API，支持 mermaid、D2、PlantUML 等十几种格式。把 mermaid 源码 POST 过去，拿回 SVG，嵌入 HTML。

这个方案很轻量。remark 插件几十行代码，没有本地二进制依赖，构建时间回到约 20 秒。Kroki 的服务端 mermaid 版本比 CDN 版本稳定，文字裁切问题少了很多。

但也发现了一个 mermaid 本身的问题：**颜色不可预测**。Kroki 用 mermaid 的默认主题渲染，节点有不同的填充色——蓝的、绿的、橙的。亮色模式下还好，暗色模式下 invert 反色后，这些颜色被反转成各自的互补色，图变成五颜六色的。

解决办法是加 `saturate(0)` 去色再 invert。效果是对的——纯黑纯白，没有任何彩色——但流程上多了一步"先让 mermaid 生成带颜色的图，再用 CSS 把所有颜色洗掉"。

本质上，颜色不可控不是因为 Kroki，而是因为 mermaid 的主题系统：`themeVariables` 参数能覆盖大部分颜色，但不同节点类型有独立的配色槽位，逐一覆盖很繁琐；`themeCSS` 用 CSS 注入，但 mermaid 某些内置样式用 `!important` 或更高优先级的选择器，偶尔盖不掉。

**为什么没选**：mermaid 的样式控制是系统性问题，换个渲染环境只是换了问题出现的时机。

## 方案四：D2

第四个想法是换个图表语言。

D2 是 Terrastruct 开发的声明式图表语言。和 mermaid 最大的区别在于，D2 把"图表的结构"和"图表的样式"彻底分开。mermaid 对你说："你写图的结构，样式用我的主题系统来控制"——然后你发现主题系统不够用。D2 对你说："你写图的结构，样式用 CSS 来控制"——然后你发现它真的就是 CSS。

一个 D2 的流程图长这样：

```d2
direction: right

用户目标 -> Agent Loop: 持续推动任务
Agent Loop -> 上下文工程: 决定模型看见什么
Agent Loop -> 工具系统: 连接模型与真实世界
工具系统 -> 权限与安全: 约束副作用
上下文工程 -> Agent Loop
工具系统 -> Agent Loop
```

语法比 mermaid 直觉：箭头用 `->` 而不是 `-->`，方向用 `direction: right` 而不是 `graph LR`。节点名直接写，不需要 `[]` 括号。标签跟在箭头后面，不需要 `|text|` 语法。

D2 的渲染由 `d2` CLI 完成，输出是干净的 SVG。Kroki 也支持 D2——所以可以用同一套 Kroki API，换成 `d2/svg` 端点。

我们最终的选择是：用 Kroki 同时支持 mermaid 和 D2，现有 mermaid 图不用改，新图用 D2 写。构建时间保持在约 20 秒，样式完全由 CSS 控制，不依赖任何图表语言的内部主题。

## 四种方案对比

| | 客户端 mermaid | Playwright | Kroki + mermaid | Kroki + D2 |
|---|---|---|---|---|
| 渲染质量 | 文字裁切 | 完美 | 较好 | 较好 |
| 样式可控 | 弱（依赖 mermaid 主题） | 中（themeCSS） | 弱（mermaid 内部主题） | **强（纯 CSS）** |
| 构建时间 | ~12s | ~55s | ~20s | ~20s |
| 本地依赖 | 无 | 500MB Playwright | 无 | 无 |
| 新图语法 | mermaid 独有 | mermaid 独有 | mermaid 独有 | D2 + mermaid 兼容 |
| 维护成本 | 高（浏览器兼容） | 高（基础设施） | 低 | **低** |

## 为什么说 D2 更先进

mermaid 是 2014 年开始的项目，它的设计出发点是"让 JavaScript 渲染流程图"。它成功了——mermaid 是 GitHub、GitLab、Notion 等平台的标准图表语言。

但 mermaid 的架构有一个历史包袱：它既是图表语言的解析器，又是渲染引擎。这两个角色耦合在一起，意味着每加一个新图表类型（mindmap、timeline、block diagram），就要在渲染引擎里加一套新的布局算法。结果是功能越来越多，但核心的样式控制和文本渲染质量，一直没有根本性的改善。

D2 的架构不同。它把图表编译和渲染分开：D2 编译器把 `.d2` 源文件编译成中间表示（IR），渲染引擎负责把 IR 画成 SVG。这种分离带来了几个直接的好处。

**样式和结构分离。**mermaid 的主题系统是一个封闭的变量表——约 30 个 `themeVariables`，每个控制一种颜色。D2 用 CSS 类控制样式——图表中的任意元素都可以通过类名选择器精确定位。这不是"更灵活的配置"，是根本不同的抽象层次。mermaid 给你一个调色板，D2 给你整个 CSS 引擎。

**语法更干净。**`direction: right` 比 `graph LR` 更接近自然语言。`A -> B: label` 比 `A -->|label| B` 更简洁。subgraph 用花括号表示层级，不需要 `subgraph` / `end` 关键字。语法上的简化为零（写 D2 不会比 mermaid 更短），但语义更直观——不需要记忆 mermaid 特有的关键字和分隔符。

**工具链完整。**D2 有一个 Go 写的 CLI，一个 VS Code 插件（实时预览 + 自动补全），一个格式化器（`d2 fmt`），一个布局引擎调试工具（`--watch` 模式）。mermaid 的工具链相对碎片化——渲染靠 `mermaid-cli`、预览靠在线编辑器、格式化不存在——因为它的设计中心是浏览器运行时，不是命令行工具。

**布局更合理。**D2 的布局引擎（基于 ELK）在处理大型图时比 mermaid 的 dagre-d3 产生更紧凑、更可读的布局。对于 5-10 个节点的小流程图差异不明显；对于 20+ 个节点、有多层嵌套 subgraph 的图，D2 的布局质量差距就会显现。

这些差异共同指向一个判断：mermaid 是为"在网页上画图"设计的，D2 是为"用代码描述图"设计的。前者的核心场景是浏览器内的即时渲染，后者的核心场景是文本到图的编译管道。当你的需求是"在文章里嵌入干净的图表"——这是编译管道的场景，不是浏览器渲染的场景。

## 我们选的路线

最终的方案不是"全部换成 D2"，而是一个混合策略：

- 构建时通过 Kroki API 同时支持 mermaid 和 D2 两种格式，用同一个 remark 插件处理
- 现有 mermaid 图不用改，继续跑
- 新图用 D2 写，享受更干净的语法和更强的样式控制
- 亮暗模式通过 CSS `saturate(0)` + dark mode `invert(1)` 处理，不需要图自身感知主题
- 没有任何本地依赖，构建时间约 20 秒

这个选择的核心判断是：**图表工具的选择不应该是"一刀切"的迁移**。mermaid 有庞大的存量生态和用户习惯，D2 有更好的设计和工具链。让两者共存，各自发挥优势，比强迫选一个更务实。

## 最后

写这篇文章的过程中，我在四个方案之间反复切换，每次切换都带着"这次应该对了"的期待，然后被下一个问题打回来。回头想，这个折腾的过程本身就是选择判断的一部分：不是每个"看起来能用的方案"在真实场景下都经得起考验。

如果你也在给自己的博客或文档系统选图表方案，这里有两条最省时间的建议：

如果图表数量少、样式要求不高、不想引入任何依赖——直接用 Kroki + mermaid，十分钟配好。

如果想要长期的维护性、干净的样式控制、以及面向未来的图表语言——从 D2 开始。语法学习成本半小时，回报是之后每次加图都不需要和渲染引擎搏斗。
